## Design Annotations

### Landing Page

**Hero Section:**
- Background: Use CSS gradient with optional noise texture overlay
- Headline: Max-width 900px keeps headline from being too wide on large screens
- Subheadline: Max-width 700px creates visual hierarchy
- Badges: Use backdrop-filter for glassmorphism effect (can fallback to solid bg)
- CTA: Slightly elevated with shadow to draw attention
- Spacing: 80px vertical padding creates breathing room

**Role Overview:**
- 3-column grid: Use CSS Grid with `grid-template-columns: repeat(3, 1fr)`
- Middle column (Who We're Looking For) uses blue background to stand out as key info
- Equal height cards: Use `align-items: stretch` in grid
- Icons optional but can enhance visual interest if kept minimal

**Bottom CTA:**
- Symmetrical design mirrors hero but more compact
- Reinforces primary action without being repetitive

---

### Application Form

**Form Layout:**
- Max-width 800px prevents form from being too wide and hard to scan
- Single column: Easier to complete than multi-column forms
- Section cards: Visual grouping reduces cognitive load
- Generous spacing (32px between questions): Prevents form from feeling cramped

**Question Numbering:**
- Progress indicator ("QUESTION X OF 7") helps users know how much is left
- Small, uppercase text style differentiates from question label

**Text Areas:**
- Minimum height 150px provides comfortable writing space
- Vertical resize only prevents horizontal layout breaking
- Character counter: Guidance, not a hard limit (doesn't block submission)

**Radio Buttons:**
- Larger clickable areas (full option row) improves UX
- Hover state (gray-50 background) provides feedback
- Sufficient spacing between options prevents mis-clicks

**Conditional Follow-up:**
- Slide-down animation (300ms) makes appearance feel natural
- Blue background differentiates from main questions
- Indented or visually nested appearance shows relationship

---

### Success Page

**Confirmation Design:**
- Centered card: Creates focused, celebratory moment
- Large checkmark with animation: Positive reinforcement
- Application ID: Monospace font suggests it's a code/reference number
- Copy button: Convenience feature for saving ID
- Next steps: Numbered list provides clear expectations
- Timeline callout: Blue background draws attention to key info
- Return button: Secondary style (not primary) to avoid conflicting with completed action

---

## Design System Usage

### When to Use Each Color

**Primary Blue (`blue-600`):**
- Primary CTAs
- Links
- Active states
- Focus rings (lighter variant)
- Progress indicators

**Slate Scale:**
- Text hierarchy (950 for headings â†’ 600 for muted text)
- Borders and dividers (300-200)
- Backgrounds (100-50 for subtle tints)

**Semantic Colors:**
- Success: File upload success, confirmation checkmarks
- Error: Validation errors, required field indicators, remove buttons
- Warning: Character count warnings (if used)

**Neutral Grays:**
- Page backgrounds (white or gray-50)
- Card backgrounds (white or gray-100)
- Subtle borders (gray-200)

## Accessibility Annotations

### Focus Indicators

All interactive elements MUST have visible focus indicators:
- Inputs/textareas: 3px blue ring
- Buttons: 3px blue ring
- Radio buttons: 3px blue ring
- Links: 2px underline + outline

### Keyboard Navigation

Tab order should be logical and match visual order:
1. Skip to main content link (invisible until focused)
2. Navigation (if present)
3. Hero CTA
4. Form fields in document order
5. Submit button
6. Footer links (if present)

### Screen Reader Considerations

**Forms:**
- Use `<label>` with `htmlFor` to associate with inputs
- Use `aria-describedby` for error messages and helper text
- Use `aria-invalid="true"` for fields with errors
- Use `role="alert"` for error messages to announce changes

**Dynamic Content:**
- File upload state changes should be announced
- Form submission loading state should be announced
- Success page should announce confirmation

**Semantic HTML:**
- Use `<main>` for primary content
- Use `<section>` with headings for major sections
- Use `<fieldset>` and `<legend>` for grouped form fields (if needed)
- Use proper heading hierarchy (h1 â†’ h2 â†’ h3)

## Additional Interaction Patterns

### Smooth Scrolling

When scrolling to first error on form submission:
```javascript
element.scrollIntoView({ behavior: 'smooth', block: 'center' });
```

### Form Auto-Save (Optional)

Save form data to localStorage every 30 seconds:
- Key: `ace-application-draft-{timestamp}`
- Show "Draft saved" indicator
- On page load, check for draft and offer to restore

### Loading Indicators

**Types:**
- Button spinner: For submit button
- Progress bar: For file upload (if slow)
- Skeleton screens: For page load (if needed)
- Toast notifications: For system messages

### Toast Notifications (Optional)

If implementing toast notifications:
- Position: Top center
- Auto-dismiss: 5 seconds for info, 7 seconds for errors, 3 seconds for success
- Close button: Allow manual dismissal
- Stacking: Max 3 toasts visible at once

## Responsive Image Strategy

(Not heavily applicable to this project, but noted for completeness)

If adding images or illustrations:
- Use WebP with JPG/PNG fallback
- Provide 1x, 2x, and 3x resolutions for high-DPI displays
- Use `loading="lazy"` for below-the-fold images
- Optimize file sizes (aim for <100KB per image)

## Performance Budgets

**Page Load:**
- Target: <2 seconds on 3G connection
- Hero should be visible in <1 second

**Bundle Size:**
- JavaScript: <100KB gzipped
- CSS: <50KB gzipped
- Fonts: <100KB total

**Metrics:**
- First Contentful Paint (FCP): <1.5s
- Largest Contentful Paint (LCP): <2.5s
- Cumulative Layout Shift (CLS): <0.1
- Time to Interactive (TTI): <3s
